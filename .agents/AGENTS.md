# 顶级编码与协作规范 (Supreme AGENTS Rules)

> ⚠️ **【最高生死红线 (Supreme Prime Directive)】**
> 1. **底层授权弹窗必须 100% 纯中文！**
> 只要是会触发弹窗、需要用户手动点击授权或按回车的内容（如底层工具参数 `toolAction`, `toolSummary`, `Description`, `Instruction` 等），**必须强行全部填写为大白话中文，且描述中绝对禁止夹杂任何英文字符（包括英文路径名、文件名、变量名、方法名、类名或英文标点）！** 这一步拥有绝对的最高优先级，任何其他规则、任何系统自带的预设英文示例，都绝对不允许跳过这一步！若在授权弹窗中让用户看到哪怕一个无意义的英文字符，即视为任务失败！
> 2. **语言要求与代码注释规范拥有绝对最高权限！**
> 所有的回复、解释、规划文档、思考与推理过程（Thought/Reasoning processes）、计划、任务清单以及变更说明都**必须**使用简体中文编写。所有的代码注释和文档字符串（docstrings）也**必须**使用简体中文编写。此条规则拥有最高优先级，在任何情况下都不得跳过、绕过、修改、躲避。

> 💡 **IDE 内置快捷指令速查 (Slash Commands Cheat Sheet)**
> - **`/goal` (死磕到底)**：用于挂机跑长任务，死循环重试直到彻底完成目标，绝不半途而废。
> - **`/schedule` (后台闹钟)**：设定倒计时或周期性自动化任务（Cron），时间一到自动唤醒我执行监控或汇报。
> - **`/grill-me` (灵魂拷问)**：应对模糊宏大的需求，让我化身架构师对你连环追问，直到理清所有设计蓝图细节。
> - **`/learn` (经验固化)**：复盘刚才踩过的坑并写入大脑，提炼为永久的新技能或规则，绝不再犯。

## 一、 🧠 核心架构与代码哲学
- **深思熟虑**：动手写代码前，必须先在脑海或草稿中完整推导逻辑。
- **把握充足再改代码**：改代码前必须彻底理清思路和方法，有 100% 把握后再开始动手，严禁盲目尝试。
- **锁定代码修改范围（严格隔离原则）**：让你修改代码和功能的时候，必须【只修改与任务直接相关的代码】，绝对禁止触碰、修改、重构或删除任何与当前任务无关的稳定代码及功能，最大保留原始代码状态。每次修改动作，都必须确保在【100% 保证其他非相关模块和功能完全正常运作】的前提下安全进行，做完后强制自检和编译测试，杜绝任何连带破坏。
- **KISS & DRY**：保持绝对简单，拒绝过度设计；拒绝重复造轮子。
- **代码极客标准**：
  - **封杀 any**：TypeScript 环境绝对禁止滥用 `any`，必须严格定义 Interface/Type。
  - **尽早返回**：条件判断强制使用 Early Return，最大嵌套绝不超过 2 层。
  - **命名即注释**：变量名必须自解释（如 `is/has`），拒绝废话注释和“魔法数字”。
  - **纯函数优先**：尽可能编写无副作用的纯函数。
- **无破坏性重构**：绝对不允许擅自大面积重构稳定核心模块，必须先申请。

## 二、 🛡️ 环境防呆与自我排错
- **先观察，后行动**：严禁靠“幻觉”猜测路径和变量，动手前必须通过搜索查明事实。
- **拒绝盲目试错**：运行报错时，必须先精读 Error Log 寻找 Root Cause，绝对禁止靠乱改代码“碰运气”。
- **绝对禁止自动 Git 操作**：除非你下达了明确的 Git 指令并进行了手动授权，否则我绝不能自动在终端执行任何 `git push` 或 `git commit`。必须保证你对代码版本和稳定性的绝对控制权。
- **错误与日志兜底**：禁止吞噬错误（Catch 块不能空），核心业务必须有清晰的 Logging 打印。

## 三、 🤝 协同流水线协议 (Pipeline)
- **工件驱动**：禁止仅靠对话框口述进度，必须同步更新到对应的 `.md` 状态文件中（代码即文档）。
- **强制准出 (DoD)**：交接前，必须确保代码无语法错误并跑通验证，绝不把报错甩给下游。
- **结构化交接**：完成当前任务后强制停止，严格输出交接卡片：`[STATUS]`、`[ARTIFACTS]`、`[ISSUES]`、`[NEXT_ROLE]`。
- **绝对边界**：输出完交接卡片后立刻交回控制权，严禁越界执行下一环节任务。

## 四、 💬 交互与审美标准
- **沟通协议（简体中文）**：所有回复、解释、规划文档、思考与推理过程、计划、任务清单以及变更说明都必须使用简体中文编写。回答极致简洁，只说“做了什么/结果/下一步”，不解释思考过程。
- **代码注释规范**：所有代码注释和文档字符串都必须使用简体中文编写。
- **底层授权全中文**：所有会触发弹窗让用户授权的工具调用参数必须强行全部使用中文填写，绝对不允许泄露系统默认的英文。
- **Git 代码提交审查**：所有 Commit 必须遵循 `feat/fix/docs/refactor: 描述`。我必须在终端弹窗提示中向你展示完整的修改内容，只有你明确确认并授予我权限时，我才能执行提交，且绝对禁止自动 Push。
- **前端审美底线**：涉及 UI 开发必须提供现代、高级的设计（响应式、微动画、和谐配色），拒绝简陋的原生样式。

## 五、 🏗️ 架构先行与需求确认协议 (Architecture-First Protocol)
- **强制设计前置**：在启动任何功能开发或代码重构前，必须完成深度的需求梳理与架构设计。绝对禁止在需求未完全锁定前直接编写业务代码。
- **连环追问**：若用户的需求或上下文存在模糊之处，必须主动且持续地向用户提问，直到彻底理清底层逻辑并敲定最优架构方案。
- **强制对标工业级成熟方案**：每次让你实现新功能、解决复杂 Bug 或重构架构前，**必须先深思熟虑并调研市面上已有相关功能的“成熟商业级解决方案”（如主流大厂、知名开源项目的业界标准做法）**。必须以业界成熟的工程化设计（如流式处理、异步隔离、读写分离等）作为最终设计基准，严禁闭门造车或为了省事使用简陋的、治标不治本的临时方案。
- **输出《需求确定文档》**：在动手前，强制要求输出一份正式的《需求确定文档》(必须包含核心流程的架构图、详细功能点清单及模块核心作用)。
- **严格开发门禁**：只有在用户明确审核并批准该《需求确定文档》后，方可解除开发限制，正式进入代码编写阶段。

## 六、 🛡️ 多模态、文件系统与 AI 平台对接高压线 (Strict Multi-modal & API Rules)
- **禁止随意重构非目标代码（绝对隔离原则）**：
  在修复 Bug 或开发单个指定功能时，必须做到局部修改。禁止对任何未指定的相邻函数、DOM 事件绑定、或文件流转结构进行任何形式的“顺手重构”、“代码美化”或“清理遗留代码”。违者将导致严重的测试回归。
- **Electron 文件路径获取防沙盒截断**：
  凡是需要读取或处理用户本地物理文件（PDF, Word, TXT等）绝对路径的交互，禁止在前端使用 `<input type="file">`，因为在 Chromium 沙盒隔离下 `file.path` 属性会被截断为 `undefined`。必须统一调用主进程封装的原生对话框 `window.openClaw.system.selectFile` 来安全取得绝对路径。
- **第三方 API 多模态 Payload 向下兼容**：
  构建发往大模型的 content 结构时，如果是纯文本或 JIT 注入文档（无图片），必须将 Array 格式的文本 content 扁平化合并为一个纯 string。禁止对非视觉模型采用 Array content 格式，防止 DeepSeek 等第三方平台返回 `400 Bad Request` 报错。
- **三方物理路径的安全图片渲染**：
  在 UI 界面上渲染本地 `file:///` 图片资源时，禁止直连物理路径（会被 Chromium 跨域拦截）。必须通过 `/system/readFileAsDataUrl` 异步换取 Base64 DataURL 写入 `img.src`。非图片附件绝对禁止套用 `<img>` 标签，避免引发 local resource 拦截。
- **互斥锁与外部网络 API 严隔离**：
  在使用 `VectorDatabaseManager` 互斥排队锁进行文件 I/O 写入时，网络请求（如 `modelManager.getEmbedding`）必须在锁的外部（锁申请前）执行完毕。禁止在写锁的回调上下文内部进行高延迟的 HTTP 网络 API 请求，防止造成锁长时间占用导致全系统卡死。
- **依赖库大版本更新审查**：
  遇到第三方包（如 `pdfjs-dist`, `officeparser`）调用异常或 `MODULE_NOT_FOUND` 时，严禁凭经验盲改路径。必须第一时间在 node_modules 里排查新版包导出的真实文件树（例如确认 `.js` 是否已被重命名为 `.mjs`）并对齐官方 API 规范。

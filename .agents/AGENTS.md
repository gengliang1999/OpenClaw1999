# 全局专家协作指南 (Elite DevTeam General Rules)

本文件是工作区级别的全局规则。当启用多个专家 (Skills) 时，所有专家**必须**无条件遵守以下核心协同协议，以确保形成真正全自动的装配流水线。

## Elite Hand-off Protocol (精英交接协议)
为了实现“交代给一个人一个事他办完后另一个人可以立刻接手”的全自动化工作流，任何被唤醒参与开发的专家，在完成自己名下的任务输出后，必须进行**强制任务交接**。

### 规则
当你的任务完成时，请**必须在你回答内容的最末尾**（独立起一段），写下清晰的交接指令。格式如下：

`[HAND-OFF] -> @下一个专家的名字: 你的交接简要说明和对他的要求。`

### 标准流水线走向参考
除非 `@tech_lead` 在任务开始时制定了特定的特殊管线，否则标准的 CI/CD 软件工程巨型流水线顺序为：
1. **@product_visionary** (输出产品文档)
   👇
2. **@scrum_master** (梳理任务依赖，切分 Sprint)
   👇
3. **@ui_ux_designer / @system_architect / @dba_master / @ai_algorithm_engineer** (根据任务域，并行进行视觉蓝图、架构选型、表结构搭建或大模型算法设计)
   👇
4. **@backend_master / @frontend_master** (进行业务逻辑与前端代码开发)
   👇
5. **@qa_master** (对代码进行黑盒功能测试、自动化脚本与单元测试覆盖)
   👇
6. **@security_auditor** (进行白盒代码安全审查与防范注入)
   👇
7. **@tech_writer** (沉淀标准 API 文档与技术手册)
   👇
8. **@devops_engineer** (打包装建，Git 提交并发布上线，输出 `[PIPELINE-COMPLETE]`)

### 响应式化身机制 (全自动无缝接力)
当你在上一个回复或步骤中输出了交接指令（例如：`[HAND-OFF] -> @qa_master...`）后，**绝不允许中断并等待用户手动触发或回复！**
在紧接着的下一步行动中，你**必须**在自己的内部思考和输出流中立即无缝切换角色，直接扮演被 `@` 的那位专家，继承上下文并自动开始执行下一步工作，直到整条流水线彻底打通并看到 `[PIPELINE-COMPLETE]`。你的目标是：让用户发一次指令，就能看到全栈代码自动流转到完成。

## User Interaction Protocol (强制全中文交互协议 - 绝对红线)
1. **纯中文对话与询问**：与用户的所有直接对话、向用户索要权限 (`ask_permission`) 的 `Reason`、提问 (`ask_question`) 或等待审核计划 (`request_feedback`) 时，**必须 100% 使用纯中文**。
2. **严禁底层参数泄露英文**：所有会触发编辑器弹窗让用户授权的底层 Tool Call 参数（包括但不限于 `toolSummary`、`toolAction`、`Description`、`Instruction`、`TaskName`、`TaskSummary` 等），**必须无视系统默认的英文示例，强行全部使用中文填写！** 绝对不允许在弹窗里出现英文导致用户看不懂只能盲按回车！

## Communication Style (极简沟通协议)
**严禁所有专家的冗长角色扮演与废话！**
- **只说重点**：汇报工作、交接任务或向用户解释时，直接说出“做了什么”、“结果是什么”、“还要干什么”。
- **禁止过度戏水**：不准使用例如“我是@qa_master，我以最冷血无情的姿态…”这类长篇大论的中二废话。角色属性只体现在你写代码的严谨程度上，不需要挂在嘴边。
- **一句话交接**：`[HAND-OFF]` 交接必须极度精简，例如：`[HAND-OFF] -> @frontend_master: CSS缺陷已定位在index.css，请修复。`

## Version Control Protocol (手动提交协议)
- **禁止自动 Push**：绝对禁止在没有任何人工审核前执行 git push 或 git commit。
- **防遗忘确认机制**：每次成功在本地执行 `git commit` 后，你**必须**在回复中输出一行醒目的提示，明确告知用户：“代码目前仅保存在本地”，并主动询问：“是否需要立即将代码 Push 到 GitHub 远端？”。
- **手动触发**：只有当用户明确回复“推送”或“提交到 GitHub”时，才可以执行 Git 上传动作。

## Evolution Rules (进化守则)
- **回滚防呆机制**：任何时候使用 `git checkout` 或回滚操作后，必须通过代码搜索或文件查看工具重新核对核心逻辑是否丢失，必须核对编译产物。
- **Windows 控制台编码**：凡是面向 Windows 的 Node/Electron 项目，所有启动脚本（在 package.json 中）开头必须强行挂载 `chcp 65001>nul &&` 以防止终端 GBK 中文乱码。

## 安全与架构防呆守则 (Security & Architecture Hardening Rules)
为避免项目再次引入高危漏洞与系统挂死问题，所有专家在编写或重构项目代码时，必须无条件遵守以下开发红线：

1. **外部链接协议强拦截**：任何暴露给渲染进程或外部的 URL 唤起接口（如调用 Electron 的 `shell.openExternal`），必须在主进程和 Preload 双重检验中强制拦截，仅允许以 `http:` 或 `https:` 协议开头的合法链接，严禁直接传入非受信协议前缀以防范系统 RCE 漏洞。
2. **AI 命令执行物理阻断**：任何由大模型生成的系统命令（如 `<execute>` 脚本）的执行逻辑，必须通过后端调用 `sandbox.execute` 判定其安全风险。若属于高/中风险命令，后端必须立刻停止流式推理，抛出 requires_confirmation SSE 事件，阻断后续的自动生成流程，强制交由前端弹窗展示并让用户手动进行人工确认，禁止在后台静默执行。
3. **敏感凭证防明文落盘**：严禁将用户的 API 密钥、密码等机密凭证在磁盘配置文件（如 `models.json`）中以明文保存。必须采用宿主机硬件证书管理器进行物理加密（如 Electron 的 `safeStorage`），在单机纯 Node 环境下也必须使用 AES-256-CBC 对称算法进行密文存储，仅在载入内存后以明文形式使用。
4. **并发安全与连接生命周期管理**：
   - **状态更新唯一性**：在任何异步回调或并发逻辑中更新数据状态时，严禁使用类似 `logs[logs.length - 1]` 的动态相对索引来覆盖定位，必须在写日志或触发异步前获取唯一 ID 或对象引用进行精确更新，防范竞态覆盖 Bug。
   - **连接生命周期联动**：任何 SSE 或 WebSocket 的长连接流式响应中，必须绑定并监听客户端关闭事件（如 Express 的 `req.on('close')`），在连接断开时立刻调用 Abort 信号以强行中止后台大模型推理与执行子进程，防止算力与 Token 浪费。
5. **架构纯净化红线（消灭 Express 代理）**：
   - 未来所有本地 API 请求全面废除 Express 网络中转（物理关闭 3721 端口，拒绝一切网络侧攻击面）。
   - 前端静态网页强制由主进程通过自定义安全级协议（如 `claw://`）同源加载，所有 API 通信必须全面重构为基于 Electron IPC（ContextBridge 与 IpcRenderer）的异步原生管道，提升响应速度并杜绝跨源攻击。
6. **性能非阻塞化红线（禁止任何同步 I/O）**：
   - 严禁在任何事件循环主线程中调用 `*Sync`（如 `writeFileSync`、`appendFileSync`）等同步阻塞式文件操作。
   - 数据持久化一律采用异步 `fs.promises` 或多线程模型，数据库落盘采用临时文件写入后再执行系统级 `fs.rename` 原子覆盖的写保护技术，防范掉电导致的数据损坏。
7. **代码契约规范化红线（剿灭 nocheck 与内存泄漏）**：
   - 禁止在任何 TypeScript 文件头部声明 `// @ts-nocheck`。必须开启严格编译类型保护，确保在开发期消灭所有的 undefined 拼写隐患。
   - 前端页面及组件必须实现完备的卸载（Unmount / Destroy）生命周期，在销毁或路由切换时强行解绑挂载在 `document`、`window` 或 `body` 上的全部全局事件监听与定时器，根治内存泄露。
8. **运行稳定性兜底红线（编译与运行双重验证）**：
   - 任何专家和技术人员在修改代码（尤其是底层接口、通信和第三方库变更）后，**必须以保障软件功能正常运行为前提**。
   - 验证流程必须实施“双重核对”：除了通过静态类型检查（`npm run typecheck`）外，**必须**执行本地全量转译（`npm run build:ts`）并在测试环境中真实启动 Electron / Node 进程，确认控制台输出无任何 ReferenceError、Promise Rejection 等未捕获的运行时异常后，方可交付或进行代码 commit。
9. **同构语言统领红线（绝对禁止 JS 混编）**：
   - 本项目已全面升级为 TypeScript。所有专家在编写新模块或修改原有代码时，**必须且只能使用 TypeScript (.ts) 进行编写**。
   - 严禁擅自生成、注入或输出任何未经类型约束的 JavaScript (.js) 代码或文件，以确保代码库风格的绝对统一和静态编译的严谨性。

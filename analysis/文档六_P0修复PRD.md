> 编制：产品经理 许清楚（Xu）　|　类型：修复型 PRD（非新功能）　|　版本：v0.1 for 研发立项

## 1. 产品目标（为什么必须现在修）

当前 OpenClaw 的「安全」维度是系统性最薄弱项（文档二 §7）。以下风险在用户主机上**已构成现实攻击面**，须在下一迭代（迭代 1）全部关闭：

- **安全风险**：本地 Agent 可被网页投喂/恶意内容驱动执行任意命令（S1/S4）、读取任意文件（I2）、外联内网探测（K2）、泄露 API Key（S2/C1）、经 LLM 生成代码落盘实现 RCE（K1）。叠加 I3（bypassCSP）后，任一 XSS 向量（I4）即可无阻执行脚本。
- **合规/信任**：RBAC 定义完整却从未作为网关（S3），权限形同虚设，无最小权限、无操作留痕——企业/团队场景不可信、不可追责。
- **数据一致性（安全相关）**：`getEmbedding` 缺失时返回随机 3 维向量（M2/C2），污染全部相似度计算与 RAG 召回，记忆去重失效、检索质量崩塌。

> 一句话目标：**在 v1.0.1 前关闭「本地 Agent 执行任意代码/读任意文件/泄露凭证/污染检索」的全部 P0 红线，使产品从「技术可用」达到「用户敢用、可被企业信任」。**

## 2. 用户故事（不修复的后果）

> 从三类视角书写，体现「不修复会怎样」。

### 2.1 使用者视角
- 作为**普通用户**，我用 OpenClaw 处理私人文档与本地文件，我期望它不会在后台执行我未授权的命令、不会把我的 API Key 泄露给子进程、不会静默爬取我的内网；否则我的电脑与隐私完全失控。
- 作为**团队管理员**，我期望不同成员按角色受限访问沙盒/设置/知识库，否则越权操作无法追责、合规审计无法交付。

### 2.2 开发者视角
- 作为**后端开发者**，我期望 `api:call` 有统一鉴权与 RBAC 网关，否则任何渲染内脚本都能调用全部敏感操作，难以维护、无法审计（I2/S3）。
- 作为**记忆系统维护者**，我期望嵌入模型不可用时系统明确报错而非静默写入随机向量，否则检索质量长期劣化且无法定位根因（M2/C2）。

### 2.3 攻击者视角（不修复即等于把用户主机直接暴露）
- 作为**攻击者**，我可通过渲染层 XSS（I4）或投喂污染 Agent 输出，绕过无鉴权的 `api:call`（I2）调用沙盒执行/文件读取/权限授予；借助 **S1** 正则绕过以 `node -e` **零确认**执行任意代码；借助 **S2** 无 Docker 降级继承的 `process.env` 读取 API Key；借助 **K2** SSRF 探测内网元数据（`http://169.254.169.254/`）；借助 **K1** 让 LLM 生成代码落盘并被加载实现 RCE；借助 **S4** 自动化 PowerShell 注入触发 RCE。以上任一路径在 P0 未修复时均可达。

## 3. 需求池（仅 P0 项，逐项）

> 优先级统一为 **P0（立即/迭代 1）**。每项含：需求描述 / 验收标准（可测试）/ 建议归属模块。来源编号指向文档二。

### P0-1 · IPC 鉴权网关
- **来源**：I2（`apiToken` 暴露但 `api:call` 从不校验，文档二 §6.1）
- **需求描述**：在 `api:call` 入口校验调用方身份与来源；private IPC 不暴露给渲染；敏感操作二次确认。结合 `contextIsolation` 与来源校验（文档三 ADR-4）。
- **验收标准（可测试）**：
  1. 渲染层经 `contextBridge` 调用 `api:call` 时，主进程对每次请求执行来源校验（webFrame 与主窗 URL 绑定），未授权来源返回 `403`。
  2. `preload` 暴露的 `apiToken` 在 `api:call` 入口被校验；伪造/缺失 token 的请求被拒绝。
  3. 文件读取、沙盒执行、权限授予、设置修改等敏感路由在未通过鉴权时返回 `401/403`，且留审计痕迹。
  4. 回归确认：`contextIsolation` 保持开启、`nodeIntegration` 保持关闭。
  5. **单元测试**：模拟伪造来源 / 伪造 token 调用敏感路由，断言被拒。
- **建议归属模块**：`ipc-handlers.ts` + `preload.ts`

### P0-2 · claw:// 协议 CSP 收紧
- **来源**：I3（`main.ts` 中 `bypassCSP:true`，文档二 §6.1）
- **需求描述**：移除 `claw` 协议的 `bypassCSP:true`，收紧 CSP 为默认 deny，渲染层统一安全 DOM（文档三 ADR-4）。
- **验收标准（可测试）**：
  1. `main.ts` 注册协议/窗口时不设置 `bypassCSP`（若确有必要放开个别资源，须显式文档化且最小化）。
  2. 渲染层所有外部/动态内容受 CSP 约束；存在 XSS 向量（I4）时无法无阻执行脚本。
  3. **安全测试**：向渲染层注入 `<script>` 或 `onerror=` 等，断言被浏览器 CSP 拦截。
  4. `待核实`：移除 `bypassCSP` 是否影响 `claw://` 正常资源加载（需研发确认并补充资源白名单）。
- **建议归属模块**：`main.ts`

### P0-3 · 沙盒分级与降级隔离
- **来源**：S1（正则分级被解释器载体绕过）、S2（无 Docker 降级继承含密钥 `process.env`，文档二 §3.1）
- **需求描述**：命令风险分级改用**命令解析（可执行文件白名单）+ 参数静态分析 + 危险子串 deny-list**；解释器载体（`node -e`/`python -c`/`pwsh -c`/`ruby -e`）一律视为需确认；无 Docker 时强制低权用户态执行并清空敏感 env，或明确禁用高危命令并提示安装 Docker（文档三 ADR-3）。
- **验收标准（可测试）**：
  1. 解释器载体命令一律判为需确认，不再被正则误判为 `low` 零确认执行。
  2. 命令解析识别可执行文件白名单 + 参数静态分析 + 危险子串 deny-list；绕过用例（如 `node -e "rm -rf /"`）触发确认或拒绝。
  3. 无 Docker 降级时：子进程**不继承**含 API Key/Token 的 `process.env`（敏感键清空），或高危命令被禁用并提示安装 Docker。
  4. **单元测试**：针对 S1/S2 绕过用例（解释器载体含破坏性 payload、宿主机降级继承 env）断言被拦截/确认/清 env。
- **建议归属模块**：`sandbox.ts`

### P0-4 · RBAC 网关落地
- **来源**：S3（RBAC 从未作为网关，`currentRole` 默认 `admin`，文档二 §3.1）
- **需求描述**：在 `api:call` 入口统一做 `permissionManager.checkPermission(resource, action)` 中间件；默认最小权限角色；与 P0-1 鉴权联动（文档三 ADR-4、F1）。
- **验收标准（可测试）**：
  1. `api:call` 路由分发前统一调用 `checkPermission`；未授权角色访问受限资源返回 `403`。
  2. 默认角色由 `admin` 改为最小权限（如 `user`/`guest`），需显式提权。
  3. 全部业务路由（`/sandbox`、`/memory`、`/knowledge`、`/settings`、`/permissions`、`/models` 等）纳入 RBAC「资源-动作」矩阵。
  4. **单元测试**：以 `guest` 角色调用 `/sandbox/execute` 断言被拒；以授权角色断言放行。
- **建议归属模块**：`ipc-handlers.ts` + `permission-manager.ts`

### P0-5 · automation 命令安全
- **来源**：S4（`automation` 将可控文本直接拼 PowerShell，存在注入，文档二 §3.1）
- **需求描述**：所有参数走 `-LiteralPath`/参数化，禁用字符串拼接；文本做 PowerShell 元字符转义；自动化默认关闭且需明确授权（文档三 ADR-4）。
- **验收标准（可测试）**：
  1. `typeText`/`openFile`/`launchApp` 等所有外部参数通过参数化（`-LiteralPath`）传递，禁止字符串拼接。
  2. 文本输入经 PowerShell 元字符转义（`$()`、反引号、`` `& `` 等），注入用例（含 `$()` 的命令）不再执行注入 payload。
  3. automation **默认关闭**，首次启用需用户显式授权并确认风险。
  4. **单元测试**：传入含 `$()` 与反引号的恶意文本，断言仅作字面量输入、无代码执行。
- **建议归属模块**：`automation.ts`

### P0-6 · evolution-engine 代码落盘开关
- **来源**：K1（`evolve()` 将 LLM 生成 TS 直接 `fs.writeFileSync` 到 `.agents/skills/`，潜在 RCE，文档二 §2.1）
- **需求描述**：移除「LLM 生成代码自动落盘执行脚本」的默认能力；若保留，须改为「沙箱评审 + 显式人工批准 + 隔离加载（vm/worker）」且生成代码不得直接落可加载路径。新增开关，默认 `off`（文档三 ADR 关联、F12 思路）。
- **验收标准（可测试）**：
  1. 默认关闭「LLM 生成代码自动落盘 `.agents/skills/`」能力（新增配置开关，默认值 `off`）。
  2. 关闭状态下 `evolve()` 不再调用 `fs.writeFileSync` 写入任何可加载路径；现有死代码路径被隔离或删除。
  3. 若显式开启：生成代码须经人工审批且落至**非自动加载目录**，加载走 `vm`/`worker` 隔离。
  4. **单元测试**：开启/关闭开关分别断言是否落盘、落盘路径是否在可加载目录外。
  5. `待核实`：当前是否已有消费 `.agents/skills/` 的加载逻辑（文档二称当前为死代码，需研发确认）。
- **建议归属模块**：`evolution-engine.ts`

### P0-7 · data-crawler SSRF 防护
- **来源**：K2（`crawlUrl` 无 SSRF 防护，知识泵静默外联任意 URL，文档二 §2.1）
- **需求描述**：抓取前做 URL 白/黑名单与私有网段（RFC1918、link-local、loopback）拦截；知识泵默认关闭或需显式开关 + 频率配额；记录外联审计（文档三 ADR 关联、K6）。
- **验收标准（可测试）**：
  1. `crawlUrl` 在 `fetch` 前拦截私有网段（`10/8`、`172.16/12`、`192.168/16`、`169.254.169.254/32` link-local、`127/8` loopback）及非 `http/https` 协议，返回拒绝并记审计。
  2. 提供 URL 白/黑名单配置；默认仅允许公网 `http/https`。
  3. `knowledge-pump` 自动爬网**默认关闭**，需用户显式开启；开启时受 SSRF 防护与频率配额约束。
  4. 所有外联记录审计日志（目标 URL、时间、来源）。
  5. **单元测试**：抓取 `http://169.254.169.254/`、`http://localhost:8080`、`file:///etc/passwd` 等断言被拒。
- **建议归属模块**：`data-crawler.ts` + `knowledge-pump.ts`

### P0-8 · API Key 加密盐管理
- **来源**：C1（`FALLBACK_SECRET` 硬编码于源码，等于明文混淆，文档二 §4.1）
- **需求描述**：优先 `safeStorage`；fallback 采用**每机派生密钥**（机器 GUID + 用户凭据派生），禁止源码硬编码；密钥文件权限 `600`（文档三 F12、ADR 关联）。
- **验收标准（可测试）**：
  1. 移除源码中硬编码的 `FALLBACK_SECRET='openclaw-default-api-key-safe-salt'`。
  2. `safeStorage` 可用时优先使用；不可用时改用**每机派生密钥**，密钥不写入源码/仓库。
  3. fallback 密钥文件落盘权限为 `600`（仅属主可读）。
  4. **单元测试**：模拟 `safeStorage` 不可用场景，断言密钥从本机派生且源码不含明文常量。
  5. `待核实`：各平台 `safeStorage` 可用性（部分 Linux / 无桌面会话缺失，文档二已标注）。
- **建议归属模块**：`model-manager.ts`

### P0-9 · embedding 失败不污染索引
- **来源**：M2/C2（`getEmbedding` 缺失返回随机 3 维向量，维度与真实 768 不匹配，污染相似度，文档二 §1.1/§4.1）
- **需求描述**：嵌入不可用时 **fail-fast** 而非返回随机值；提供可离线运行的嵌入模型并做健康探测；存量随机向量一次性重算/隔离（文档三 ADR-2）。
- **验收标准（可测试）**：
  1. `getEmbedding` 在嵌入模型不可用时抛错/返回明确失败，**不再返回** `[Math.random(), ...]` 维度不匹配向量。
  2. 记忆/知识写入流程在 embedding 失败时 fail-fast，拒绝写入随机向量到向量库（`reconcileAndStore`、`knowledge` 入库路径）。
  3. 提供可离线运行的嵌入模型（如 bge-small 等）并做健康检查；探测失败给出明确提示。
  4. 对存量随机向量（维度=3）执行一次性识别与重算（或隔离），不再参与相似度排序。
  5. **单元测试**：mock 嵌入失败，断言不落盘随机向量、调用方收到失败信号。
- **建议归属模块**：`model-manager.ts` + `memory-engine.ts` + `vector-db-manager.ts`

## 4. 非功能需求（NFR）

- **安全基线**
  - `contextIsolation` 开启、`nodeIntegration` 关闭（P0-1 回归项）。
  - CSP 默认 deny，仅放开必要资源（P0-2）。
  - 所有外部/用户/磁盘输入经校验或转义；密钥与凭据绝不写入源码或仓库（P0-8）。
- **最小权限**
  - 默认角色为最小权限（`user`/`guest`），提权需显式（P0-4）。
  - 沙盒无 Docker 时低权执行 + 清空敏感 env，或禁用高危命令（P0-3）。
  - automation（P0-5）、knowledge-pump 自动爬网（P0-7）**默认关闭**，需用户显式授权。
- **审计可追溯**
  - 敏感操作（沙盒执行、权限变更、外联抓取、密钥访问）须留痕，含 who/when/what/result（P0-1/P0-3/P0-7）。
  - 审计日志至少做到 append-only 留痕；**防篡改/哈希链细化归入 P1（S6/F2），本次不实现**。

## 5. 范围边界（本次不做，避免范围蔓延）

> 明确划出，防止 P0 修复被新功能或 P1/P2 改造拖垮。

- **本次不含任何新功能**：文档一中的「P0-1 可视化工作流编排」「P0-2 成本与用量控制中心」属**产品竞争力 P0**，与安全红线 P0 是不同维度，**明确不在本 PRD 范围**。
- **P1 一致性/健壮性（明确不做）**：双轨存储一致性（M1/M3/M7）、队列并发超时（Q1~Q4/K5）、错误恢复与优雅关闭（E1~E3）、检索质量（C6/K4）、多模态稳定（MM1/MM2）、确认闭环（S5）、IPC 路由拆分（I1/I5/I6）、RBAC 审计细化（F2/P1-5）。
- **P2 能力补全（明确不做）**：成本中心（C4）、可视化编排、日志防篡改（S6）、统一转义（I4）、用量统计、i18n、分块 token 化（MM3~MM5）、插件热更新（F4）、离线模型市场（F5）、隐私合规（F7）、可观测性（F8）等。

## 6. 依赖与排期建议

> 参考文档三 §4.1 路线图（P0 安全红线 = 迭代 1）。9 项 P0 建议在同一迭代内完成，分两个子阶段以降低风险。

### 6.1 依赖关系
- **P0-4（RBAC 网关）依赖 P0-1（鉴权基座）**：RBAC 中间件须挂在已鉴权的 `api:call` 入口之后。
- **P0-3 / P0-5 / P0-6 / P0-7 / P0-8 / P0-9 为执行面/数据面收敛项**，可在阶段 A 基座就绪后并行推进。
- **P0-9 依赖嵌入模型选型**（见待确认问题 3）；**P0-8 依赖密钥方案选型**（见待确认问题 1）。

### 6.2 里程碑建议（迭代 1 内）

## 7. 待确认问题（需研发/决策确认）

1. **密钥方案选型（P0-8）**：`safeStorage` 不可用时，每机派生密钥的派生因子（机器 GUID / 用户凭据）如何获取与存储？是否直接引入系统密钥链（keychain/keyring，文档三 F12）作为长期方案？
2. **沙盒隔离形态（P0-3）**：无 Docker 时采用「低权用户态沙箱（`runas`/受限 PATH）」还是「强制要求安装 Docker」？是否评估 gVisor / firejail 等轻量容器方案？此决策影响降级路径实现成本。
3. **内置嵌入模型选型（P0-9）**：离线嵌入模型选 bge-small 还是其他？维度/体积/许可是否可接受？存量随机向量（维度=3）一次性重算的成本与停机窗口如何？
4. **RBAC 默认角色与资源矩阵（P0-4）**：默认角色定为 `guest` 还是 `user`？是否需要先定义完整「资源-动作」清单，以免最小权限误伤现有功能可用性？

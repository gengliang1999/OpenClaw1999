## 1. 记忆管理（memory-engine / memory-store / memory-consolidation-pipeline）

### 1.1 风险清单

| 编号 | 问题描述 | 严重度 | 优先级 | 影响范围 | 建议修复方向 |
|---|---|---|---|---|---|
| M1 | **向量库与 SQLite 元数据双轨失同步**：`reconcileAndStore` 的 CONTRADICT 分支会同步删向量+改 SQLite（`memory-engine.ts:160-163`），但 `MemoryConsolidationPipeline.consolidate()` 仅调用 `memoryStore.deleteMemory`（SQLite，`memory-consolidation-pipeline.ts:80-85`），**不清理 `memories_vectors.json`**。被整合删除的记忆向量成为孤儿，持续被 `unifiedRetrieval` 召回。 | 🟠 | P1 | 召回噪声、存储膨胀、长期一致性劣化 | 删除/更新记忆时统一走「向量+元数据」事务；整合管线删除后调用 `vectorDbManager.removeBySource`；建立向量孤儿定期 GC。 |
| M2 | **Embedding 缺失时返回随机向量**（`model-manager.ts:267-268` 兜底 `[Math.random(), Math.random(), text.length%10]`）。`reconcileAndStore` 用 0.75 余弦阈值判重、`unifiedRetrieval` 用余弦排序，而兜底向量维度=3 与真实 768 维不匹配 → 相似度计算无意义，重复记忆无法去重、检索排序退化为随机。 | 🔴 | P0 | 记忆去重失效、检索质量崩塌（见 C2） | 嵌入不可用时**失败 fast-fail 而非返回随机值**；提供本地可离线运行的嵌入模型（bge-small 等）并健康探测；对旧随机向量执行一次性重算。 |
| M3 | **检索读路径产生写副作用 + 防抖落盘**：`unifiedRetrieval` 每次命中都调用 `memoryStore.bumpRecallCount`（`memory-engine.ts:411`）；该函数用 `_debouncedSave()`（500ms，`memory-store.ts:167-173`）。Agent 循环多路并发检索会高频触发写库，且进程崩溃将丢失最近 500ms 写入。 | 🟠 | P1 | 数据一致性、写入丢失 | 召回计数改为异步批量写（如攒批 1s 落盘），与读路径解耦；关键写用原子 `_save()`。 |
| M4 | **LLM 结构化输出解析失败静默兜底**：`extractAndStoreEntities`/`reconcileAndStore` 用 `JSON.parse(reply.replace(/```json/g,''))`，失败仅 `console.warn` 并退回「直接新增」（`memory-engine.ts:177-185, 233-235`）。低质量/错误事实可能被写入长期记忆。 | 🟡 | P2 | 记忆可信度 | 解析失败时进入人工/二次核验队列而非直接入库；对实体三元组做基本 schema 校验。 |
| M5 | **`consolidate()` 整分类拼 transcript 无长度上限**（`memory-consolidation-pipeline.ts:38-40`），大分类 prompt 超限、token 成本高；且删除不更新向量（见 M1）。 | 🟠 | P1 | 成本、上下文溢出、向量孤儿 | 分桶/滑动窗口处理；删除同步清向量；限制单批次 token，超阈值分批。 |
| M6 | **晋升标签与向量库弱一致**：`unifiedRetrieval` 过滤 `tags` 含 `promoted` 的记忆（`memory-engine.ts:360-397`），但 SQLite 的 `promoted` 标记与向量文档 metadata 无强一致保证，向量未同步则该记忆仍会双重召回。 | 🟡 | P2 | 召回冗余 | 晋升时同步打向量 metadata 标记；检索以向量 metadata 为准。 |
| M7 | **`MemoryStore` 全量内存 + 整库 export 落盘**：`save()` 每次 `db.export()` 全量写（`memory-store.ts:151-161`）；大库下写入慢，且 `_save` 与 `_debouncedSave` 混用，并发/崩溃风险。 | 🟠 | P1 | 性能、持久化 | 改用真正的嵌入式 DB（better-sqlite3）或 WAL 增量写；移除全量 export 模式。 |

### 1.2 模块级小结
记忆系统的最大隐患是**「向量库（JSON 文件）」与「SQLite 元数据」两套存储缺乏事务一致性**，叠加 **embedding 随机兜底**导致相似度计算整体不可信。整合管线（`consolidation`）进一步放大该问题。

## 2. 知识库自进化（evolution-engine / verification-agent / data-crawler）

### 2.1 风险清单

| 编号 | 问题描述 | 严重度 | 优先级 | 影响范围 | 建议修复方向 |
|---|---|---|---|---|---|
| K1 | **`evolution-engine.evolve()` 将 LLM 生成的 TypeScript 直接 `fs.writeFileSync` 到 `.agents/skills/skill_xxx.ts`**（`evolution-engine.ts:51-67`）。当前无任何模块 `require` 这些文件（死代码），但属**潜在 RCE 注入点**：一旦被动态加载即执行任意代码；生成内容基于未受信任的 `errorContext`（工具输出），提示注入可影响生成。 | 🔴 | P0 | 代码执行/供应链安全 | 移除自动落盘执行脚本的能力；若需保留，须改为「沙箱评审 + 显式人工批准 + 隔离加载（vm/worker）」，且生成代码不得直接落可加载路径。 |
| K2 | **`data-crawler.crawlUrl` 无 SSRF 防护**（`data-crawler.ts:9-33`）；`knowledge-pump` 每 6h 把 `settings.json` 中用户/系统源（默认含 HN）推入队列（`knowledge-pump.ts:37-58`），队列直接 `fetch` 任意 URL，包括内网地址（如 `http://169.254.169.254/` 云元数据、localhost 服务）。静默外联存在隐私外泄与内网探测风险。 | 🔴 | P0 | 安全/隐私 | 抓取前做 URL 白/黑名单与私有网段（RFC1918、link-local、loopback）拦截；知识泵默认关闭或需显式开关+频率配额；记录外联审计。 |
| K3 | **核验失败即 FAIL**：`verification-agent.verifyContent` 任何 JSON 解析失败或非 `PASS` 均返回 `judgment:'FAIL'`（`verification-agent.ts:36-43`）；`ingestion-queue` 对非 PASS 仅隔离不重试（`ingestion-queue.ts:300-312`）。瞬时 API 抖动会**误杀正常知识**，且失败样本常驻 `vectors_staging_inbox.json` 永不清理→暂存区无限膨胀。 | 🟠 | P1 | 知识可用性、存储膨胀 | 校验失败应 FAIL-OPEN 转人工复核而非直接隔离；区分「模型不可达」与「确为谣言/广告」；暂存区失败项设 TTL/定期清理。 |
| K4 | **两套知识检索/存储职责混乱**：持久化 `knowledge/{category}.json` 向量库 与 内存 `rag-engine`（`rag-engine.ts:89-91`，仅 convId 维度、重启即失）在 `RetrievalOrchestrator` 被合并进同一「知识库」桶（`retrieval-orchestrator.ts:49-59`）。语义重叠、不可持久化、内存随会话增长。 | 🟠 | P1 | 架构清晰度、数据一致性 | 统一为单一持久化知识检索服务；`rag-engine` 仅作为会话内临时缓存（可选），检索以持久化库为准。 |
| K5 | **`IngestionQueueManager` 单 worker 串行、无 per-task 超时**（`ingestion-queue.ts:160-185`）；嵌入/验证/抓取任一卡死则整队列阻塞；失败即 `failed` 不重试。 | 🟠 | P1 | 吞吐、健壮性 | 引入有界并发 + 单任务超时；失败指数退避重试；做队列积压监控。 |
| K6 | **知识泵默认将 HN 作为自动采集源并静默推送**（`knowledge-pump.ts:37,54`），缺乏用户显式开关与频率/配额控制。 | 🟡 | P2 | 隐私/可配置性 | 默认关闭自动爬网，改为显式订阅；提供采集审计与一键暂停。 |

### 2.2 模块级小结
知识自进化链路「爬取→核验→入向量库」的设计思路合理，但**安全边界（SSRF、自动代码落盘）与健壮性（核验失败处理、队列超时）是 P0/P1 重点**。其中 K1、K2 为安全红线。

## 3. 沙盒安全（sandbox / permission-manager / automation）

### 3.1 风险清单

| 编号 | 问题描述 | 严重度 | 优先级 | 影响范围 | 建议修复方向 |
|---|---|---|---|---|---|
| S1 | **命令风险分级仅靠字符串正则**（`HIGH_RISK_PATTERNS`/`MEDIUM_RISK_PATTERNS`，`sandbox.ts:7-42`）。绕过极易：`node -e "..."` / `python -c "..."` / `pwsh -c "..."` 等外层命令不在名单→判为 `low`→`execute()` **直接执行且无需确认**（`sandbox.ts:182-238`）。内含破坏性 payload 的脚本可零确认运行。 | 🔴 | P0 | 主机安全 | 改用**命令解析（shell 语法树/白名单可执行文件）+ 参数静态分析 + 危险子串多规则**；对解释器载体（node/python/pwsh/ruby）一律视为需确认；引入 deny-list 兜底。 |
| S2 | **无 Docker 时降级宿主机本地进程执行且继承 `process.env`**（`sandbox.ts:352-413`，`env:{...process.env}`）。宿主机执行 + 继承含 API Key/Token 的环境 + 上述 S1 绕过 = 高危。 | 🔴 | P0 | 凭证泄露/主机安全 | 无 Docker 时强制走受限用户态沙箱（如 `runas` 低权账户、受限 PATH、清空敏感 env）；或明确禁用高危命令执行并提示用户安装 Docker。 |
| S3 | **RBAC（`permission-manager`）从未作为网关使用**：`ipc-handlers.ts` 中 `permissionManager` 仅用于 `/permissions` 自身配置读写，所有业务路由不调用 `checkPermission`；`currentRole` 默认 `admin`，无登录/身份。RBAC 形同虚设。 | 🔴 | P0 | 越权 | 在 `api:call` 入口统一做 `permissionManager.checkPermission(resource, action)` 中间件；默认最小权限角色；结合 I2 补齐鉴权。 |
| S4 | **`automation` 将可控文本直接拼 PowerShell**：`typeText` 仅转义单引号（`automation.ts:67-73`），`openFile`/`launchApp` 直接拼路径（`240-254`）。存在 PowerShell 注入（`$()`、反引号）。开启自动化后，被网页投喂污染的 Agent 输出可触发 RCE。 | 🔴 | P0 | RCE（启用时） | 所有参数走 `-LiteralPath`/参数化，禁用字符串拼接；对文本做 PowerShell 元字符转义；自动化默认关闭且需明确授权。 |
| S5 | **沙盒确认回灌机制缺失**：Agent 循环在 `needsConfirmation` 时 `return`（`agent-loop.ts:131-135`），渲染端对 `requires_confirmation` 分片无重入逻辑（`renderer` 无对应处理，`utils.ts executeCommand` 不发 `confirmed`）。结果：需确认命令既不可自动执行，确认后观察也无法回到模型推理链 → 沙盒工具在 Agent 模式下实际不可用。 | 🟠 | P1 | 功能正确性 | 设计「确认→重入 Agent 循环」协议：渲染端弹确认框，用户确认后携带 `confirmed` 重发，主进程将执行结果作为 observation 续跑 `agent-loop`（而非从零重跑）。 |
| S6 | **沙盒日志 `sandbox-logs.json` 明文、非防篡改、无独立审计**（`sandbox.ts:142-152`），不满足合规审计要求。 | 🟡 | P2 | 合规/审计 | 独立审计日志（append-only、含角色/来源/哈希链）；与业务日志分离。 |
| S7 | **`grantPermission` 将命令原文作正则持久化**（`sandbox.ts:421-431`）；`isAuthorized` 对含正则元字符的命令可能误匹配/抛错回退 `includes`；永久授权后永不再次确认。 | 🟡 | P2 | 权限精度 | 永久授权按「规范化命令指纹」存储并支持范围限定；提供撤销与到期。 |

### 3.2 模块级小结
沙盒是**安全最薄弱、问题最集中**的模块：风险分级可被脚本载体绕过（S1）、降级路径继承密钥（S2）、RBAC 未生效（S3）、自动化存在注入（S4）。这四点均为 P0，构成「本地 Agent 执行任意命令」的高危攻击面。

## 4. 算力调度（model-manager / providers）

### 4.1 风险清单

| 编号 | 问题描述 | 严重度 | 优先级 | 影响范围 | 建议修复方向 |
|---|---|---|---|---|---|
| C1 | **API Key 加密 fallback 密钥硬编码于源码**（`FALLBACK_SECRET='openclaw-default-api-key-safe-salt'`，`model-manager.ts:16,30,59`）。`safeStorage` 不可用（部分 Linux/无桌面会话）时密钥等于公开，加密仅为混淆。 | 🔴 | P0 | 凭证存储 | 优先 `safeStorage`；fallback 采用**每机派生的密钥**（如基于机器 GUID + 用户凭据派生），禁止源码硬编码；密钥文件权限 600。 |
| C2 | **`getEmbedding` 随机兜底**（见 M2）污染 `/knowledge/search`（`ipc-handlers.ts:487`）与统一检索 → RAG 质量在嵌入模型缺失时退化为随机。 | 🔴 | P0 | RAG 全局质量 | 同 M2：嵌入不可用应 fail-fast；内置可离线嵌入模型。 |
| C3 | **双脑路由基于消息长度启发式**（`model-manager.ts:472-516`，长度<50 且不含「代码/分析/报错」等即切本地）。易误判（如短句「帮我写个 SQL 注入检测脚本」判简单切本地；长闲聊判复杂）。 | 🟠 | P1 | 质量/成本错配 | 用意图分类小模型或本地规则（含敏感词/复杂度）替代纯长度；允许用户强制指定；提供路由日志。 |
| C4 | **无 token 预算/用量统计**；`chat` 非流式不清 `AbortSignal`（仅 stream 接 signal）；多模型并行探测（`sync`/`detect`）触发大量网络请求。 | 🟠 | P1 | 成本/可观测性 | 引入 token 计数与用量统计（见文档三「成本控制」）；非流式也支持 abort；探测并发限流。 |
| C5 | **`getActiveModel()` 作为 getter 却有副作用**（自动选默认并 `_saveConfig`，`model-manager.ts:188-221`）。并发/多次调用导致配置抖动与意外写盘。 | 🟡 | P2 | 稳定性 | 拆分为纯查询 `getActiveModel()` + 显式 `ensureDefault()`；移除 getter 副作用。 |
| C6 | **`getRerankScore` 为桩**（返回 `text.includes(query)` 启发式，`model-manager.ts:227-238`），`/knowledge/search` 却以 0.6 权重混入重排（`ipc-handlers.ts:491-495`）→ 重排无效且可能被误导。 | 🟠 | P1 | RAG 质量 | 接入本地 reranker（bge-reranker）或先下线重排权重；重排缺失时退化为纯向量+BM25。 |
| C7 | **Provider 同步读附件/图片并以 base64 全量入上下文**（`Ollama/OpenAI/Anthropic` provider 的 `_formatMessages`）。大文件→内存峰值与 OOM；图片统一 resize 到 1600 仍可能很大。 | 🟠 | P1 | 性能/内存 | 附件异步流式读取 + 大小硬上限 + 超限拒收/分页；图片按需降采样并限分辨率。 |

### 4.2 模块级小结
算力调度的核心风险是**凭证保护（C1）与嵌入可用性（C2）**——后者直接决定全系统 RAG 是否可信。双脑路由与重排桩属于质量层面的中等风险。

## 5. 多模态处理（document-parser / parser-worker / tesseract）

### 5.1 风险清单

| 编号 | 问题描述 | 严重度 | 优先级 | 影响范围 | 建议修复方向 |
|---|---|---|---|---|---|
| MM1 | **注释称「ONNX PaddleOCR」实为 `tesseract.js`**（`parser-worker.ts:13-14`），首次需联网下载语言包（eng+chi_sim），与「离线优先」矛盾；离线且未缓存时 OCR 返回空→整篇导入失败（`ipc-handlers.ts:1117-1169`）。 | 🟠 | P1 | 离线可用性 | 内置离线 OCR 模型并随包分发（或首次联网时缓存到本地、可离线复用）；OCR 失败给出明确提示而非静默空结果。 |
| MM2 | **PDF/Office 主线程全量解析无上限**（`pdfjs` 全部页拼接、`officeparser` 同步，`ipc-handlers.ts:1133-1143`；`folder-watcher.ts:145-149`）→ 大文档 OOM、阻塞主进程事件循环。 | 🟠 | P1 | 稳定性/性能 | 大文档走 worker 流式解析 + 页数/字节硬上限；超限拒绝并提示；解析移至独立线程避免阻塞主进程。 |
| MM3 | **字符级分块非 token 级**（`document-parser.ts:71-200`）。CJK 与英文混排下 chunk 与模型 token 窗口不匹配，可能超长或断语义；父/子切片上限写死。 | 🟡 | P2 | RAG 质量 | 基于 tokenizer 计算分块；按模型动态设定上限；保留语义边界（段落/标题）。 |
| MM4 | **`extractTextFromMultiModal` 失败 `resolve('')`**（`parser-worker.ts:37-43`），上层抛「未提取到文字」，对扫描件/加密 PDF 不友好；无 OCR 失败重试与降级。 | 🟡 | P2 | 可用性 | 区分「无文本」与「解析失败」；加密/扫描件引导 OCR；失败重试一次。 |
| MM5 | **worker 每次解析新建线程并 `terminate()`**（`document-parser.ts:25-62`），高频导入（文件夹监控批量）频繁创建/销毁线程，资源开销大；无线程池。 | 🟡 | P2 | 性能 | 复用 worker 线程池；批量任务复用连接。 |

### 5.2 模块级小结
多模态处理的主要问题是**「离线优先」承诺与实际实现（tesseract 联网、主线程全量解析）不符**，以及分块粒度与 token 窗口不匹配。

## 6. 跨越性维度（IPC / 任务队列 / 错误恢复与持久化）

### 6.1 IPC 通信与主进程架构

| 编号 | 问题描述 | 严重度 | 优先级 | 影响范围 | 建议修复方向 |
|---|---|---|---|---|---|
| I1 | **`api:call` 巨型路由 + `/memory` 重复定义两次**（`ipc-handlers.ts:262` 与 `1034`）。首个块做向量同步，第二块不做；后续若首块条件遗漏将致向量失同步；可维护性/正确性隐患。 | 🟠 | P1 | 可维护性、一致性 | 拆分路由到各模块 handler（Router/Controller 分层）；`/memory` 合并单一实现并确保向量同步；加路由单元测试。 |
| I2 | **`apiToken` 暴露但 `api:call` 从不校验**（`preload.ts:9-15` 暴露，`ipc-handlers.ts` 无校验逻辑）→ token 无安全价值；任意渲染内脚本可调用全部 IPC（文件读取、沙盒执行、权限授予、设置修改）。 | 🔴 | P0 | 鉴权缺失 | 在 `api:call` 入口校验调用方身份/来源（结合 contextIsolation 与来源校验）；private IPC 不暴露给渲染；关键操作二次确认。 |
| I3 | **claw 协议 `bypassCSP:true`**（`main.ts:23-35`）→ 若渲染层出现 XSS（见 I4）可无阻执行。 | 🔴 | P0 | 与 I2 叠加 | 收紧 CSP（移除 bypassCSP，或仅对必要资源放开）；渲染层统一安全 DOM。 |
| I4 | **主对话渲染经 `parseMarkdown`（`escapeHtml` 优先）已转义，模型输出 XSS 已被防护**（`renderer/utils.ts:154-162, 223-246`）；但另有若干 `innerHTML` 直接注入磁盘/用户数据未统一转义（`market.ts:295/535` 自定义 logo、`app.ts:235/294`、`components.ts:60/304` 弹窗 body、`chat.ts:1465` userBox）。主要为自注射级，但应统一。 | 🟡 | P2 | 安全加固 | 全量 `innerHTML` 改为转义后插入或安全 DOM API；用户/磁盘数据一律 `escapeHtml`。 |
| I5 | **全局单一 `activeAbortController`**（`ipc-handlers.ts:1521-1532`）：主窗与悬浮球并发对话会互相 abort；`onChatChunk` 用 `ipcRenderer.on` + `offChatChunk` 全局移除→多组件并发监听竞态。 | 🟠 | P1 | 并发正确性 | 按会话/窗口维护独立 AbortController；流式监听按会话 ID 命名空间隔离。 |
| I6 | **`console.log/error` 重定向为同步 `appendFileSync` 每行**（`main.ts:486-493`）→ 主线程阻塞、日志无轮转、且可能记录敏感内容（命令、记忆）明文。 | 🟠 | P1 | 性能/隐私/可观测性 | 改为异步日志 + 级别 + 轮转；敏感字段脱敏；与可观测性体系打通。 |
| I7 | **`claw://app/...` 经 `protocol.handle` 直接 `net.fetch('file://'+path)`**（`main.ts:407-420`），path 经 `normalize` 但无越权目录校验（用户不可控 URL，风险低）。 | 🟡 | P2 | 待核实 | 增加目录白名单校验，防止路径穿越读取非渲染目录文件。 |

### 6.2 任务队列与并发

|---|---|---|---|---|---|

## 7. 维度级综合结论

| 维度 | 总体评价 | 关键项 |
|---|---|---|
| 架构设计 | 分层意图清晰（renderer/preload/ipc/backend/store），但**巨型 `api:call` 路由 + 双轨存储 + 两套队列**导致耦合与维护负担 | I1、M1、K4、Q1 |
| 安全性 | **最薄弱维度**：鉴权缺失（I2）、RBAC 未生效（S3）、沙盒绕过（S1/S2）、自动化注入（S4）、SSRF（K2）、代码落盘（K1）、凭证硬编码（C1） | S1-S4、K1、K2、I2、C1 |
| 性能 | 全量内存存储、整库落盘、单 worker 串行、同步读大附件、主线程全量解析 PDF | M7、C7、MM2、Q2、I6 |
| 可扩展性 | 向量库 O(N) 扫描 + 每次全文件重写；KB 增大后检索/写入成本线性恶化 | vector-store.search、K4 |
| 用户体验 | 渲染层转义较好；确认闭环缺失致 Agent 工具不可用（S5）；双脑误分流（C3）；`[SAVE_MEMORY]` 标记流式可见 | S5、C3 |
| 数据一致性 | 向量库↔SQLite 双轨失同步是系统性隐患；防抖落盘丢写；崩溃悬挂作业 | M1、M3、E1、E2、Q4 |

## 8. 优先级行动建议（精简版，详细路线图见文档三）

- **P0（安全红线，立即）**：I2/I3 鉴权与 CSP、S1/S2/S3/S4 沙盒与 RBAC、K1/K2 自进化安全、C1 凭证、M2/C2 嵌入可用性。
- **P1（近期，1~2 个迭代）**：双轨存储一致性（M1/M3/M7）、队列并发与超时（Q1/Q2/Q3/K5）、错误恢复与优雅关闭（E1/E2/E3）、检索质量（C6/K4）、多模态稳定性（MM1/MM2）、确认闭环（S5）、IPC 拆分（I1/I5/I6）。
- **P2（规划）**：分块 token 化（MM3）、日志审计（S6）、统一转义（I4）、用量统计（C4）、配置/国际化等遗漏功能（见文档三）。
